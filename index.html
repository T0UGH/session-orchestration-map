Human-Controlled Multi-Session Orchestration

一个大需求如何被拆成多 Session 推进

这不是抽象方法论海报，而是一张来自真实实践的编排图：人作为主控，把跨仓大需求拆成多个 session，交给 Claude Code、Codex、Mira 与人工步骤共同推进，并在多层 review、实现和验证回环中逐步收敛。

Layer 3
Agent / Task Split
从 design 收敛切换到执行编排：跨仓设计被拆成多个执行单元，不只按 repo 拆，也按专项能力拆，同时承认有些任务必须由人亲自做。

Layer 4
Agent Plan Loops
每个 agent 都不是接任务直接开干，而是先跑一套 plan / review 小回环。reviewer 不只是提建议，而是阶段出口判断者。

Claude Code Plan

Planner 产出：plan

每个执行单元先用 Claude Code 出一版 plan，而不是直接实现。

Codex Review Plan

Reviewer 异构 review

由 Codex 来 review plan，形成和 planner 不同的视角。

Planner 不接受点 ↔ 人机讨论

Human in the Loop 产出：修正版 plan

planner 会明确说哪些不接受，再和人讨论这些分歧，讨论后修正 plan 并继续回环。

Plan Revision

迭代修正

不是一次性 review，而是多轮 plan 修正与再 review。

Reviewer Gate

Codex decides 出口判断

是否收敛、是否可以继续往下走，由 reviewer 判断，而不是 planner 单独决定。

Loop B：Claude Code plan → Codex review → planner 说哪些不接受 → 人与 planner 讨论 → 修正 plan → reviewer 判断是否放行
Layer 5
Implementation Loop + Global Reviewer
reviewer 放行后，agent 直接进入实现。但实现本身又形成第二组回环；单 agent 局部收敛后，还要进入更高层的整体 reviewer 回环。

New Codex Coding Session

Codex 产出：代码实现

进入实现时，会新开一个 Codex session 专门写代码。

Coder Summarizes Reviews

汇总 review 反驳机制

写代码的 Codex session 负责汇总 review，并决定吸收哪些意见、反驳哪些意见，再继续修正代码。

5 路 reviewer 并行审查

Code Revision

吸收 / 反驳 / 修正

不是 review 来一条改一条，而是 coder 统一处理，再决定如何进入下一轮。

Global Reviewer

System-level 整体看三个 agent

局部收敛后，还要站在整体视角检查三个 agent 的结果能否拼起来，继续跑整体级回环。

Loop C：新 Codex session 写代码 → 多 reviewer 并行审查 → coder 汇总 → 反驳 / 吸收 → 修正代码 → 再循环
Loop D：跨 agent 的整体 reviewer 从全局视角继续回环，专门检测边界问题、系统级问题、局部正确但全局不成立的问题。
Layer 6
Verification / Conditional Return
整体 reviewer 放行之后，流程还不会停在 review，而是进入真实验证层。发现问题时，不是固定回到某一层，而是按问题性质回流。

Integration

联调

跨 agent 成果开始真正拼接。

Testing

测试

验证局部与整体行为是否符合预期。

Real Run Validation

实跑验证

真正把系统跑起来，看流程是否闭环。

Conditional Return Router

按问题性质回流

问题可能回到实现 agent、Global Reviewer、planner，必要时甚至回到更早的 design 层。

这是一版单 HTML 原型：重点在于先把层级、回环、角色关系和条件回流讲清楚，而不是把所有真实节点塞满。下一版如果继续迭代，可以把每层节点做得更细、把连接线画成更强的可视化网络，并加入适度交互高亮。
